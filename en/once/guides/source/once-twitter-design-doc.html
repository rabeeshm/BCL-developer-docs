Twitter Amplify VMAP Design Document

Jeffrey Young, Jesse Michael

Introduction

The Twitter Amplify VMAP (TAV) is an extension to the Once UX product that will create a new VMAP format that Twitter is prepared to consume in their Twitter client. A Once UX request from the Twitter Amplify Player (TAP), or when specifically requesting the Amplify VMAP, must return an Amplify VMAP response. The system will be able to follow the correct business logic for a single Once URL to conform to either a TAV VMAP response or the standard Brightcove VMAP. Project specification and requirements can be found in more detail at Twitter Amplify VMAP 1.0

Requirements

Brightcove and Twitter have been working closely together to break ground on the design of this product. Twitter Amplify VMAP 1.0 must include the follow product requirements.

  • Extend the Once UX API feature set to include output of Twitter Amplify VMAP 1.0 (TAV 1.0) consumable content and ad creative metadata.
  • Detect a client making a request for content via the Twitter Amplify Player (TAP) to ensure a TAV 1.0 compatible response.
  • Creative and tracking VAST metadata to be included in the VMAP ad break elements per the VMAP specification.
  • External tracking beacons configured within the Once system to be passed back in the TAV within the applicable Twitter extension elements, with the client electing to fire all tracking URLs, not the Once system.
  • The TAV must return the Twitter Owner ID from the Once Customer and the Twitter Brand ID of the Ads if present.

Features

The Twitter Amplify VMAP requirements will be met by implementing the following features:

  • The Once Ad Service (OAS) VMAP Handler will look for the TAP Http header or a specified query string parameter to determine if it needs to build a TAV instead of a Brightcove VMAP.
  • The TAV will be built by the OAS VMAP Handler to meet Twitter's specified XML format.
  • Twitter Amplify VMAP will expose external beacons in the response that the TAP will consume and will not be consumed server side. 

Architecture

The Twitter Amplify VMAP product will be an extension of the existing Once Ad Service product. TAV requests will be formatted the same as current VMAP requests. The VMAP Handler will be responsible for determining if the VMAP response will be a TAV or Brightcove VMAP and then build the appropriate VMAP response.

Components

VMAP Handler

The VMAP Handler will be responsible for looking at the HTTP headers and checking the query string for a TAV request, then redirecting to the appropriate builder for either a TAV or a Brightcove VMAP. The HTTP header will look for the string value of " ___ ". The query string that the VMAP Handler will look for will be "forcetav". Once UX will also need to be updated to accept SSL requests, while keeping the mandatory SSL feature intact. If mandatory SSL is enabled, we should reject any request to the VMAP response with a 403 that is not secure.

VMAP Builder 

The existing VMAP builder will need to be updated to be inherited by the Twitter Amplify VMAP Builder. This will also allow the flexibility of the VMAP to be used for future client side applications.

Twitter Amplify VMAP Builder

The twitter Amplify VMAP builder will construct the Twitter Amplify VMAP response with specific twitter extensions and namespace declarations.

  • The TAV requires a <tw:content>  attribute of "ownerId" specific to the client. The Twitter Content Owner ID is the numeric ID associated with the twitter username that owns the content. This value must be configurable between various items within the same domain. This value should be set as a query string parameter, that overrides a content Extended Metadata value, that overrides an application config using the key "twownerid". The desired use case would be for the owner ID to be sent as extended metadata of the asset and other options for the purpose of failover and testing. If a content owner ID is not found we will, by default, send an empty string and proceed.
  • The TAV requires a <tw:content> attribute of "contentId" that will be the foreign key of the content being requested. 
  • The TAV requires a <tw:content attribute of "stitched" to determine if the content has the ads stitched into it. This will be forced to true for any Once link.
  • The TAV requires a media file similar to the content URI derived from the standard VMAP Builder. This will be the Once URL constructed by the VMAP Builder that contains all AD definitions in the OAS Id and has a TTL of 600 before they expire. A bool value specifying that the request originally came from the TAP will be stored in memcache and use the visit guid as a key. The ONCE API should be checking memcache using the visit guid key to disable server side beacons and ad impressions from being fired on the server side.
  • The <MediaFile> tag will have 2 attributes, Content Length and Payload Length. Content length will be floating point duration of the content itself, and payload length will be the floating point duration of the stream with an aggregated duration of the ads and content. These values should be controlled through the domain configs to allow either integer values or floating point values with configurable decimal accuracy.
  • Twitter VMAP expects external analytic tracking client beacons similar to that of VAST to be fired by the Twitter Amplify client. 
    • New client beacon types will be created that will be beacons sent to the Twitter VMAP TAV. In compliance with VAST, the beacontemplate names will need to be converted to the standardized convention of "Start", "FirstQuartile", etc.
      • ClientStart
      • ClientFirstQuartile
      • ClientMidpoint
      • ClientThirdQuartile
      • ClientComplete
      • ClientQuartile
      • ClientPause
      • ClientResume
      • ClientRewind
    • The ClientPause, ClientResume, and ClientRewind should be treated as VAST impression beacons that should always be ignored by the API unless the VMAP builder is requesting them.
    • By default, the request from the TAP should not fire the beacon types specified as content quartiles. The decision to not fire regular content quartiles (If from Twitter) can be overridden by a query string parameter of "umbp" or application config of "ForceServerBeacons". 0 or false will keep the beacons disabled server side, 1 or true will allow the beacons to fire anyway.
  • Since Twitter does not plan on supporting Smart XML in V 1.0, but our clients use it, the slot and video view impressions should be converted to content start beacons to allow the Twitter client to consume them for customer MRM tracking.
  • Since we are expecting the Twitter client to fire off all of the ad tracking and ad impression urls, these tracking event will not fire from the Once API by default (If from Twitter). This can be overridded by the existing query string parameter of "umtp" or application config of "is_impression_firing_enabled". 0 or false will keep the ad impressions disabled server side, 1 or true will allow the ad impressions to fire anyway.
  • Twitter Amplify VMAP also requires the VMAP:AdBreak to be formatted in accordance to the VAST 3.0 xsd. The only exception in this case is the <MediaFile> in which we should not output. Modifications to the VMAP:AdBreak should use the exisiting workflow of XSLT's to manipulate the desired output of the ad response.
  • Twitter Amplify VMAP can also explicitly be converted to JSON or JSONP via the query string if a lighter response is desired or future implementations of client side JSON parsing.

Once API

The Once API will need to be updated to not fire specific Content Beacons types or Ad Impressions if the request came from a TAV. The Once API will acknowledge that the request came from a TAV from the key stored in memcache. The default workflow of leaving the beacons and ad tracking to be fired by the client can be overridden by query string parameters or application configs (See Twitter Amplify VMAP Builder component) .

WOPR

A WOPR tool for creating Beacon Templates should be created to easily create and configure beacons.
