1      INTRODUCTION

1.1   Overview

This guide provides the necessary information to implement the Unicorn Once UX™ ad service within your workflow to obtain an ad server response in the form of a usable Once VOD™ URL. The functionality described in this document is only available to clients who have the API feature enabled.        

 

This document assumes prior general knowledge of API architecture and XML language. Under that assumption, it was written for Unicorn Media customers and their internal developers and integrators.

 

1.2   Terms

Commercial Ad Break

  • A predetermined point in an asset stream where a marketing message of a certain length is injected into the publisher’s video content

 

Domain

  • The top level administrative designation for clients within the Unicorn Media Platform
  • Clients can have multiple domains, but can only access one domain per login session
  • Domains contain the client Analytics, Channels, Catalogs, Users, Ad Providers, Category Lists, Publication Rules and Rights

 

Foreign Key

  • Referential value matching a field value in one database table to a field value in another table
  • Used to cross-reference tables for proper asset delivery
  • In practical use this value is most often a customer’s unique video content identifier

 

GUID

  • Globally Unique Identifier
  • A unique 128-bit number that is produced by the REST API to identify a particular component, application, file, database entry and/or user

 

m3u8

  • A file format that stores multimedia playlists specific to Apple HTTP Live Streaming (HLS)

 

VAST 3.0

  • ·         Video Ad Serving Template
  • A universal XML schema for serving ads to digital video players
  • Describes expected video player behavior when executing VAST-­formatted ad responses

 

VMAP 1.0

  • ·         Video Multiple Ad Playlist
  • An XML template that video content owners can use to describe the structure for ad inventory insertion when they don’t control the video player or the content distribution outlet

 

1.3   Dependencies

Unicorn Once UX™ relies on Unicorn Media Once VOD™ technology and therefore requires that you have an account setup. If you are unsure of your account status, please contact customer service at support@unicornmedia.com.

2      UNICORN Once UX™ DESCRIPTION

The Once UX™ is a pre-configurable web method, based on Unicorn Once VOD™ technology that allows you to make a content request and receive a single normalized response, including all ad information, in a defined XML Schema; along with your video content and in-stream video ads.

There are two request methods available through the Once UX™ that provide seamless control and functionality regardless of the type of content you store within the Unicorn Media system.

 

2.1   VAST Ad Proxy Method

This method behaves as the name indicates; an ad server proxy. Using the Once UX™ URL construct that is pre-defined to elicit a VAST response, you would create a URL that utilizes pre-configured ad logic organized to make a request to a third party ad server. The response you get back is normalized to a VAST 3.0 or SmartXML document. VAST XML contains a location called <mediafile> where the URL for the actual media file is held. The system parses the VAST 3.0 document for a foriegnkey or mediaitemGUID within <mediafile> to determine if it has already been ingested into the Unicorn File System. If the URL exists, it is replaced with a Unicorn Once™ URL that is the Ad Asset and the response is returned to the requesting client. However, if the file is not currently in the Unicorn File System, ingest and processing is initiated via Ad Cache (see the Unicorn Once Technical User Guide for more information on Ad Caching).  An empty VAST 3.0 response is returned until the ad video is ready for delivery via Ad Cache.

 

Therefore, if the application is a flash player, iPhone app, Android app, etc., you have a searchable VAST response to utilize however you wish. In addition, you have a Unicorn Once™ URL that utilizes Once VOD™ Automatic Device Detection technology for the ad asset.

 

This method also supports using HTTP cookies to handle tracking session for ad frequency capping.

 

2.2   VMAP Ad Aggregation Method

With this method, Once UX™ retrieves a multiple ad playlist from values that are derived by summarizing or aggregating the ad viewing data based on your application (player, iPhone, etc.) from appropriate ad servers. This Once UX™ URL template is pre-defined to elicit a VMAP 1.0 response providing control over ad inventory insertion where you don’t control the player application.

 

You will create a Once™ URL that sends a request to Unicorn Media servers for content. The data returned is a list of all the ads for that content viewing session normalized into VAST 3.0 and then embedded into a VMAP 1.0 XML object. This VMAP object contains a Unicorn Once™ XML object, which includes Unicorn Once™ data and a URL.  See the sample response example.

NOTE:  Be aware that the VAST 3.0 object for each  creative in the VMAP object will be missing a download link because the URL provided in the Once™ XML object will incorporate the ad resources at the correct locations causing the video ad creative content URL to be unnecessary.

The greatest difference between these two methods is that, in addition to the Once™ URL, the Ad Aggregation method also provides metadata and a VMAP object containing the ad load for the entire asset. This gives you greater client side flexibility with advertising while still using Unicorn Once™ platform. For instance, you can decide to manage tracking client side or through Unicorn Services, include banners and click-throughs not supported by Unicorn Media, etc.

3      URL CONSTRUCT CONSIDERATIONS

Unicorn Once UX™ utilizes a proprietary URL syntax based on Once VOD™ technology. You configure it through a programmatic URL generation method customized for your specific application.   

 

3.1   URL Templates

VAST Ad Proxy Method

Once UX.unicornmedia.com/now/ads/proxy/{deliveryType}/{RequestedFileType}/{domainGUID}/{applicationGUID}/[{mediaItemGUID}/][?param1=value[&param2=value]]

 

VMAP Ad Aggregator Method

Once UX.unicornmedia.com/now/ads/vmap/{deliveryType}/{RequestedFileType}/{domainGUID}/{applicationGUID}/[{mediaItemGUID}/][?param1=value[&param2=value]]

 

 

 

3.2   Information Gathering

Several pieces of information are needed to take advantage of the Once UX™ URL construct. Resulting Once™ URL(s) within responses will follow Once VOD™ convention for {deliveryType} and {RequestedFileType} based on the Once UX™ request syntax decisions you make as listed below. (See Once VOD™ Implementation Guide for more information)

  • Delivery Type: Decide if your media will be ‘stitched’ for progressive download, ‘adaptive’ for HTTP streaming or ‘od’ (on demand) to be used with Auto Links
  • Requested File Type: Choose from ‘mp4’, ‘3gp’, ‘m3u8’ or ‘auto’ which determines delivery method according to client capabilities
  • DomainGUID, ApplicationGUID, MediaItemGUID: Each are obtained by contacting Unicorn Media Customer Support
  • Query String Parameters: Use the following pages to determine your parameters

 

Some of the query string parameters will be passed through and appended to the resulting Once™ URL while others will be used by Once UX™ when making calls to ad servers.

4.2   Serving Content Variables

There are four acceptable DeliveryType/RequestedFileType combinations for the Once UX™ service; od/auto, adaptive/m3u8, stitched/3gp and stitched/mp4. Unicorn Once™ automatically delivers one or multiple bit rate profiles based on device detection; however, you can force video to play through certain profiles by temporarily overriding the standard profile list.

 

NOTE:  MP4 and 3GP file types will hit all impressions/tracking event URLs at once so if you are tracking hit data, it’s detailed by asset rather than by ad. However, m3u8 file types have playlist associated so each URL is hit as it is viewed providing tracking granularity.

 

Auto Delivery

The DeliveryType/RequestedFileType combination for auto delivery is od/auto, where ‘od’ stands for ‘on demand’. Using od/auto will allow the system to determine the appropriate profiles to include within the .m3u8 playlist using Unicorn device detection technology.

Server Side Beaconing

UMBEPARAM{key}={value}

 

This parameter adds beaconing-related data to the Once UX™ URL so that when your content is played the appropriate third-party beaconing events are fired to your beaconing server/service. A beaconing server base URL is associated to an Application GUID within the Unicorn Media database and therefore not viewable. You can, however, dynamically insert values to the beaconing URL's via the query string. The beacon fires on a playback start event. Each variable in a beaconing URL can be specified according to preferences and passed dynamically to the beacon call via the query string parameter UMBEPARAM, statically mapped and inserted within the UM system, dynamically inserted via DAP (dynamic ad parameters) or any combination of these 3 options.

 

URL Map: /…}/{domainGUID}/[{VirtualFileName}?[UMBEPARAM{key}=[{value}]]]

 

-         Insert Values Dynamically at Request via UMBEPARAM

Through dynamically inserting values into the query string, any information you would like passed to the beaconing server that we do not already have on our system (static mapping or DAP) will be sent. 

 

Example Key/Value pairs:

video=true

csid=sports

 

Once UX™ URL Example

http://Once UX.unicornmedia.com/…/?UMBEPARAMvideo=true&UMBEPARAMcsid=sports

 

Beaconing URL Example

http://beaconserver.customer.com/…/?video=true&csid=sports

 

The beacons will fire at the same instance the ‘Play’ metric is recorded with safeguards in place to prevent multiple ‘play’ events skewing the analytics. Beaconing URLs are always fired. However, the URL they fire to is dependent on what is preconfigured in the database.

 

The static key/value pairs in the database provide values that can be extended with runtime values by utilizing UMBEPARAM and DAP if desired. If the values are not passed in via the UMBEPARAM, the values will be pulled from the mapping database. This behavior is in place for all three handlers; auto, stitched and adaptive. 

 

Beaconing URLs do not require additional variables. However, the variables must be included if you want to use the UMBEPARAM query string parameter. You may also have multiple beaconing URLs per application.

 

NOTE: Each Key/Value pair specified in the UMBEPARAM must be present in the database and assigned to an application.

 

 

4.6   Query String Parameters Affecting Ad Server Request Behavior

Get Specified Ads Returned Based on Runtime Targeting

UMADPARAM{Key}={Value}

 

Use this parameter to have your ad provider serve back ads according to your requirements. The key-value pair(s) will be passed along to the ad server call.  Each variable in an Ad Request URL can be specified according to preferences and passed dynamically to the beacon call via query string parameter UMADPARAM, statically mapped and inserted within the UM system or dynamically inserted via DAP (or any combination of these 3 options).

 

Example Key-Value pairs:

cc=US

state=Arizona

 

URL Created by Customer:

http:// Once UX.unicornmedia.com/UMADPARAMcc=US&UMADPARAMstate=Arizona

 

Unicorn Media consumes the Once UX™ URL, strips off UMADPARAM and serves the key-value pair to the ad provider who in turn will serve up only the ads that were requested by the certain value.

 

URL Sent by UM to ad provider:

http://adserver.adprovider.com?cc=US&state=Arizona

 

The Once UX™ URL example above would make the ad call and return ads specific to the country and state values provided, as determined by your ad server configuration and campaign.

 

 

Append Custom Parameters

UMPTPARAM{Key}={Value}

 

This is used to pass key-value pairs into the ad server call when the value should not be URL decoded by the service. In contrast to UMADPARAM, this parameter lets you pass a string directly through the system without manipulation by Unicorn.  For example, you have the ability to pass a string value of comma delimited key-value pairs that require URL Encoding.

 

 

Example Key-Value pairs:

Cust_params=Denver%Android_4_0%2tile%Video&Video_param=26iu%3D%252F7646

 

URL Created by Customer:

http://.../?UMPTPARAMCust_Params=Denver%Android_4_0%2tile%Video&UMPTPARAMVideo_param=26iu%3D%252F7646

 

The above example depicts the parameter variables in a URL with ad targeting. Through appending an Once UX™ URL with the UMPTPARAM query string parameter, the encoded string value can be specified and passed through to the ad server without manipulation.

 

Below is a sample of the call that Unicorn will be expected to make to ad providers in order to retrieve an Ad asset. The response to the Video Ad will be in in the form of a VAST XML.

 

URL Sent by UM to ad provider:

http://adserver.adprovider.com?Cust_Params=Denver%Android_4_0%2 tile%Video&Video_param=26iu%3D%252F7646

 

 

Ad Frequency Capping

Umsessionkey={string}

 

Umsessionkey is used to store sessions for ad frequency capping for devices that don’t use cookies. By using it as a query string parameter, you can control how often customers see pre-roll ads. The current default is a user will see 1 ad per X number of content play starts. This parameter is not required for all Once™ URLs and Frequency Capping must be configured by Client Services.

 

Frequency capping can be set up on a per application GUID basis, functions through the use of cookies and is manually configured by Unicorn Media Client Services. The function affects the behavior of server side ads (ad campaigns), yet will only affect the pre-roll ad segment.

 

If your delivery and application model does not utilize cookies you may override the behavior. The value of this parameter should be a unique identifier, preferably in the form of a GUID, which your company would create and supply to UM. This unique ID is subsequently used to create a unique viewer session to enable the frequency capping behavior.

 

There are two value settings that can be configured. You will need to specify both:

  1. How many plays per ad
    1. Defines how many play starts a user should have per 1 ad start during a session
    2. Session time to live (TTL) in seconds
      1. The TTL associated with plays per ad defines the max duration of the viewer session before the plays per ad frequency is reset as though it were the first play

 

NOTE: Users playing in a frequency capped application will ALWAYS see an ad on the first play.

 

 

 

 

 

Disable Tracking Events

UMTP={integer}

 

UMTP is used to override or disable the Unicorn server side ad video impression and event tracking default behavior.

  • UMTP=0

 

By default, Unicorn Media fires linear ad video impression and tracking events to the ad server on behalf of the client/application. If the parameter is explicitly set, UM does not fire the impression or tracking event.

 

 

Output

output={integer}

 

This Key-Value pair controls the object format of the response Unicorn Media returns.

  • XML=0
    • JSONP=2
    • JSON=1

 

 

 

NOTE: When using output=2, the ‘callback’ parameter is always required.

 

 

Type

type={integer}

 

This Key-Value pair controls the type of the response Unicorn Media returns.

  • VAST=0
  • SMART XML=1

 

NOTE: This Parameter is ONLY Applicable with the Ad Proxy Method.

 

 

Callback

callback={string}

 

This Key-Value pair is the callback method name to use when writing out the JSONP response.

 

NOTE: System default is ‘?output=0&type=0’.  If those are the desired format and type it is not necessary to add parameters to the query string.
