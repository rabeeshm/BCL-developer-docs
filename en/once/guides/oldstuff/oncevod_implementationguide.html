Brightcove Once VOD Implementation Guide

Last Revision Date: Mar 2014

Overview

This guide provides the necessary information to implement Brightcove Once VODTM monetization technology with your video on demand workflow. Some of the functionality described in this document is only available to clients who have the API feature enabled. Please contact your Brightcove representative if you are experiencing any difficulty getting started.

Prerequisites

This document assumes prior general knowledge of API architecture and XML language. Under that assumption, it was written for Brightcove customers and their internal developers and integrators.

Glossary of Terms

CNAME
Short for Canonical NAME
  • A DNS database record that indicates the host name of a computer with which its aliases are associated
Channel
  • Channels are a mechanism used to maintain the publishing rules for displaying your content
  • They allow for publication rules to be assigned to assets, playlists, players and syndication feeds
Commercial Ad Break
A predetermined point in an asset stream where the main content is paused for a marketing message of a certain length
Domain
  • The top level administrative designation for clients within the Brightcove Platform
  • Clients can have multiple domains, but can only access one domain per login session
  • Contain Analytics, Channels, Catalogs, Users, Ad Providers, Category Lists, Publication Rules and Rights
Foreign Key
  • A referential value that matches a field value in one database table to a field value in another table
Used to cross-reference tables for proper asset delivery
GUID
Globally Unique Identifier
  • A unique 128-bit number that is produced by the REST API to identify a particular component, application, file, database entry and/or user
m3u8
A file format that stores multimedia playlists specific to Apple HTTP Live Streaming (HLS)

We notify you when your media items are ready for publishing as each Profile is published by using a Gateway Notification Service configured to make HTTP POST calls to any site of your choosing. Process

This service will send a notification (in a URL encoded format) on each of our Delivery Profiles, which are the conditioned versions that we create for the OnceTM platform. Requirements

Syntax

Brightcove Once VODTM utilizes a proprietary URL syntax. You configure this URL through a programmatic URL generation method that you customize for your particular business needs. This URL employs the Brightcove syndication engine to detect viewer devices, distribute content and insert and target ads according to your specifications.

5.1 Information Gathering

Several pieces of information are needed to take advantage of the Once VODTM URL construct including:

  • Host Name: The Once VODTM URL (once.unicornmedia.com) or a CNAME pointing to the host name
  • Delivery Type: Decide if your media will be 'stitched' for progressive download, 'adaptive' for HTTP streaming or 'od' (on demand) to be used with Auto Links
  • Requested File Type: Choose from 'mp4', '3gp', 'm3u8' or 'auto' which determines delivery method according to client capabilities
  • DomainGUID, ApplicationGUID, MediaItemGUID: Each are obtained by contacting Brightcove Customer Support
  • Virtual File Name: Select an arbitrary name with a file extension representing the requested file type
  • Ad Parameters: Decide what your parameters will be with regard to ad position, placement and the Ad GUID
    • Optional and only utilized if UMAID ad parameters are used, as described in this document
  • Query String Parameters: Use the following pages to determine your parameters

Server side ads

Once VODTM serves up and distributes ads through ad campaigns associated with your Application GUID to dynamically request the payload from your Ad Server(s). Advertising calls are made server-side on behalf of the client to the ad server configured within your Brightcove account. Our platform supports VAST XML, SmartXML and nearly any VAST-like XML format. Contact Brightcove Client Services and provide your VAST ad tag or FreeWheel tag URL for integration.

If the Ad Campaign, tied to that ApplicationGUID, has an associated Ad Server, Once VODTM will send a request to the Ad Server to retrieve the Ad payload. The Ad Server returns XML with data that is used to ingest and cache the ad unit within your Brightcove account. When the same ad unit is encountered for subsequent consumers, the system will utilize the cached ad video along with the unique impression and tracking events for that request.

Ad Frequency Capping

Brightcove provides ad frequency capping to let you control how often your customers see pre-roll ads. The current configurable options provide that a user will see 1 ad per X number of play starts. Frequency capping can be set up on a per application player basis. The function affects the behavior of server side ads (ad campaigns) and will only affect the pre-roll ad segment.

Requirements

Frequency capping functions through the use of cookies and is manually configured by Brightcove engineering staff. However, if your company does not utilize cookies you may use the query string parameter "umsessionkey". The value should be a unique identifier, preferably a GUID, which you would create and supply as a query string key-value pair on the initial Once VODTM URL request. This unique ID is then used to store the frequency capping data for that unique client.

Ad Unit Waterfall Method

Brightcove OnceTM supports ad creative waterfall behavior for both VAST XML and SmartXML. Once VODTM makes an ad request to the first node in sequence and, if an ad unit is returned, the ad is rendered with appropriate tracking. If an ad unit is not returned, Once VODTM will try the next node in sequence and continue until all nodes have been tried.

With this methodology, an ad creative in the sequence will be used if the ad server response is not empty and includes a valid media file.

Server Side Beaconing

UMBEPARAM{key}={value}

This parameter adds beaconing-related data to the Once VODTM URL so that when your content is played the appropriate third-party beaconing events are fired to your beaconing server/service. A beaconing server base URL is associated to an Application GUID within the Brightcove database and therefore not viewable. You can, however, dynamically insert values to the beaconing URLs via the query string. The beacon fires on a playback start event. Each variable in a beaconing URL can be specified according to preferences and passed dynamically to the beacon call via the query string parameter UMBEPARAM, statically mapped and inserted within the UM system, dynamically inserted via DAP (dynamic ad parameters) or any combination of these 3 options.

Tracking Ads with Query String Parameters

Brightcove Once VODTM facilitates ad tracking by firing ad server-supplied impressions and tracking URLs. Each ad has its own URL and therefore the number of hits to a certain URL translates to the number of times the ad associated with it was viewed. This is accomplished by appending the Once VODTM URL with the Ad Parameter query string described above as well as two other ad defining keys.

All three parameter segments in the proper order are necessary to receive tracking information. They are: 'UMAID-a-b=e', 'UMATU-a-b=' and UMTT=a-b, where a = AdPosition, b = AdPlacement and e = AdGUID respectively. You must define each parameter value for query string keys separated by a dash ("-"), as shown in the examples below, where the variables represent the point in time in seconds during the main asset when the ads should play.

AD CACHING

We can support a variety of XML structures using our XML node mapping function that relates the incoming feed to properties in our Management Platform allowing for the feed to be in any format. Therefore, you need not modify existing feeds with content in order for our system to ingest from it.

The Once VODTM AdCache component recognizes if an ad being served to your customer is present within your account. If the ad is not in our system, Once VODTM automatically ingests and caches it.

In order for Once VODTM to recognize and ingest the ad, Content Synchronization must be configured by Brightcove Customer Service to ingest your media items and metadata data. Through our modified MRSS- style process you have the ability to ingest any feed with multiple repeating XML nodes by producing a feed for each item with all content for that item contained within that node.

There are two ways to configure the Content Sync Feed behavior to meet your needs:

  • Reference – Your content will reside on a system external to Brightcove while our Media Management Platform provides references to the media items for reporting purposes

RTSP Streaming Support

Forcing a Brightcove application (player) to utilize the RTSP streaming protocol is available via Once VODTM. When using Once VODTM with the RequestedFileType of 'auto', the device will be detected with the appropriate file format delivered to the client using RTSP protocol, rather than HTTP. This "forced streaming" function can be useful when delivering to a known set of devices that only support RTSP streaming. Each asset to be played over RTSP should be transcoded into the Brightcove.Once 3GP 320 or other applicable profile(s) for optimal playback on targeted devices.

NOTE: RTSP streaming support requires a Brightcove Customer Services to perform a configuration change on a per application (player) basis.

RTSP Streaming Protocol Support via UMPRO

It is also possible to use the UMPRO command in the Once VODTM URL to force RTSP streaming protocol. However, in this scenario, the profile delivered will be the one that would normally be delivered to the device requesting the asset via standard device detection. Note: this may not be the optimal profile for RTSP delivery on the targeted device.

Closed captions

Closed captioning is a visual, textual display of the audio being played in a video asset. Since the FCC put rules in place requiring programs that were displayed originally with closed captions on TV to also have closed captions when they are re-shown on the Internet, we have incorporated this ability into our Delivery Profiles allow our customers to be compliant with these expectations. The process flow is as follows:

  1. We ingest your asset containing an embedded EIA-608 and/or CEA-708 closed caption track
  2. We then extract the closed captioning information leaving the original asset intact and create a separate SMPTE-TT binary file
  3. We incorporate this file containing the closed-caption information into all transcoded binary assets as part of the transcoding process
  4. The SMPTE-TT file is read as a single text string and formatted into an ID3v2 "tag"
  5. The ID3v2 metadata tags are then inserted into the transcoded video as an 'atom' which looks similar to the following example:
    Example MP4 ID3v2 atom tree:
    Atom ftyp @ 0 of size: 20, ends @ 20
Atom moov @ 20 of size: 101041, ends @ 101061
    Atom mvhd @ 28 of size: 108, ends @ 136
    Atom trak @ 136 of size: 56487, ends @ 56623
        Atom tkhd @ 144 of size: 92, ends @ 236
(...) lines extracted for brevity
        Atom meta @ 89010 of size: 12051, ends @ 101061
            Atom hdlr @ 89022 of size: 34, ends @ 89056
            Atom ilst @ 89056 of size: 8, ends @ 89064
            Atom ID32 [```] @ 89064 of size: 9949, ends @ 99013
            Atom free @ 99013 of size: 2048, ends @ 101061
Atom mdat @ 101061 of size: 122525163 (^), ends @ 0

    NOTE: The line in red shows an embedded ID3v2 atom starting at an offset of 89064 bytes from the containing atom.

  6. The SMPTE-TT file is then available for you through our Player API
