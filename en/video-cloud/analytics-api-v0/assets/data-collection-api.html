Introduction

The analytics data collection API provides a means to collect significant business data for analysis and customer billing. Data is sent to the analytics system via events, which are composed of a series of parameters submitted via HTTP requests.

Event

An event represents a well-defined set of data related to a single data-point within the Brightcove platform. Examples are a completed transcode for a video or the completion of DRM. Each event is made up of the event name, the dimensions and metrics associated with the event and (optionally) any other parameters that might be significant for ad-hoc analysis (in the future).

Dimension

A dimension is a descriptive attribute of an event that can be given values from a well-defined set. Each dimension contains information about the event itself (in contrast to metrics, which contain information of the event).


Video Cloud Dimensions

  • time - Number of milliseconds since epoch (January 1 1970 00:00:00)
  • account - Account ID
  • player - Player ID
  • video - Video ID

Dimensions and Junk Data

Dimensions are intended to work against specific sets of values, like time or a video ID. If an event contains inappropriate data for a dimension, it will make the overall dataset large and unwieldy and sometimes result in unusable data.

For example, if a developer mistakenly sent video size instead of video ID for the video dimension of the transcode event, all those invalid events (which the analytics system cannot detect) would severely limit the utility of the entire transcode event dataset. No reports could be generated based on video ID.

Metrics

Metrics are distinct from dimensions in that they represent specific measurement associated with the event (like the length of the video). Metrics can be either additive or gauges. An additive metric, like seconds of video transcoded or the number of times an application has been installed, accumulates over time. A gauge represents a measurement at a specific point in time, like how much gas is in your gas tank right now, or the number of videos being origin-served on December 1, 2012.

For example, given a transcode event:

http://metrics.brightcove.com/tracker?event=transcode_complete&domain=videocloud&time=1355349137201&account=4231&video=10101009&bytes=54907&seconds=30&quality=hd&priority=high

This event has the following dimensions and metrics:

  • Dimensions:
    • time The video was transcoded at 1355349137201 (Dec 12 2012, 1:51pm PST). The time dimension allows transcode event data to be rolled-up over day, week, month, year or any arbitrary timeframe.
    • account The video was transcoded for account 4231. This dimension is used to group transcode events by account.
    • video Video 10101009 was transcoded. This dimension is used to group transcode events by video. This could be used to answer "Which video was transcoded most often?"
  • Metrics:
    • bytes 54907 bytes were transcoded. This metric is used for billing the number of bytes of transcode. It can be rolled-up by dimension: How many bytes did account 4231 transcode last month?
    • seconds 30 seconds of video were transcoded. This metric can be used for billing on the number of seconds transcoded (as opposed to, or in addition to, the number of bytes).
    • quality The video quality was HD. The quality is a metric used for billing, as HD videos can be billed at a different rate from SD videos
    • priority The job has a high priority, that is should be scheduled and completed as quickly as possible (and at higher cost).


Defining New Events and Reports

While this document defines the current set of Video Cloud and App Cloud, it is expected that new events (and associated roll-up reports) will be added regularly.

In order to define new events and reports the following steps are suggested.

  1. Read this document.
  2. Understand and document the different dimensions and metrics that you need added.
  3. Open a JIRA ticked agains Reporting and Analytics to add the configuration. The team may suggest improvements and will make the requisite changes.
  4. Once the configuration is added to the analytics system, the reports should be available from the analytics API.


Additional Parameters

The documented dimensions and metrics for each event (see below for event details) are required to successfully analyze the event. In addition to those, an event may also contain any other name-value pairs that might be potentially useful in the future. For example, in addition to the required "seconds_transcoded" on the transcode event, details like the duration the transcode itself or the average bit rate of the video could also be included:

http://metrics.brightcove.com/tracker?event=transcode_complete&domain=videocloud&time=87654321&account=4231&video=10101009&bytes=54907&seconds=30&duration=16

Note These additional name-value pairs will be persisted with the event but will not be available through the reporting API by default. Additional rollup configurations can be created to make these available via the reporting API.


Sending Business Events

Single Event

To specify a single event, the event and all the associated parameters can be supplied on the query string:

http://metrics.brightcove.com/tracker?event=transcode&domain=videocloud&time=87654321&account=4231&video=10101009&bytes=54907&seconds=30

Batch Events

To send a collections of events (one or more), the format is an upload of small JSON objects, containing the associated parameters, separated by a new line.

@POST /events

{event="transcode_complete", domain="videocloud", time=87654321, account=4231, video=10101009, bytes_transcode=54907, seconds_transcode=30, priority_transcode=20}

{event="drm_complete", domain="videocloud", time=87654321, account=4231, video=10101009, bytes_drm=54907, seconds_drm=30, priority_drm=50}

{event="total_origin_served_bytes", domain="videocloud", time=87654321, account=4231, total_bytes_origin_served=54907542}

{event="total_archived_bytes", domain="videocloud", time=87654321, account=4231, total_bytes_archived=87543642}

Response Types

The response to an analytics data collection API request is denoted by a HTTP response code and a human-readable message.

HTTP Status Code

Description

Example

200

The request was successfully received by the collector and has been persisted.

{code: 200, message: "success"}

400

The request sent by the client is invalid

{code: 400, message: "Invalid 'event' parameter"}

401

The request is valid but not authorized.

{code: 401, message: "Request in unauthorized"}

50x

This is error code indicates a problem on the server side. Retry your request at another time.

{code: 500, message: "Server-side failure, please retry."}

Timeouts and Retries

It is advisable that the clients consider the following as useful strategies to deal with timeouts.

  • General connection timeout should be set to 1s.
  • Retries should be done with an exponential back-off to avoid unnecessary traffic during downtime.
  • An optional parameter, client_request, may contain a unique identifier for the event that can be used to assure no duplication.
