Unique user tracking

Unique user tracking feature is useful in identifying the number of unique browsers ( which may roughly equate to actual unique users ) sending events. This means if the same browser was to send 2 video_view events, they would be recorded as a single unique entity.

The users can be tracked by two means:

  1. A Server side browser cookie
  2. Client side unique identifier


Opt-in

User tracking is enabled only when the _track parameter is set to any value other than false. This includes the case where no value is set, ie, if the player sends an event with _track=, we would still not track the user.


Server side identifier cookie

This is the most common and simplest use case. If the player sends a request to the analytics collector with the _track flag set. The server will generate a cookie with name _bc and value set to a UUID. This cookie would be set with a max-age of 365 days after which a new cookie would be set.

  1. The max age of the cookie is reset on every visit to 365 days.
  2. If the tracking is disabled (by sending _track=false) the collector would simply not continue to set/reset cookies. This implies that the cookies that are already set would continue to exist on the viewer's browser until they are cleared.


Client side identifier

In certain cases, such as native mobile apps or certain browsers (cookies disabled), it may be difficult or impossible to accept/set/store cookies. In this scenario, the user/client application may create a unique identifier (such as a device UUID) and send it as a the user_id query parameter.
