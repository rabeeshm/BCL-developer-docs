Unique user tracking integration

  • To enable tracking the client application or player should include cookies_enabled=true parameter in the GET request to the analytics events collector.
  • To disable cookie tracking, include cookies_enabled=false or not pass a cookies_enabled parameter in the GET request.
  • The collector will now include a Set-Cookie header with the cookie __bcma with a unique identifier for the user.
  • In cases where it is not possible to set/store cookies, the client application may send back an override user identifier ( such as the device UUID ) as the user query parameter.
  • If a user parameter is specified, we WILL track the user.




Unique user tracking

Unique user tracking feature is useful in identifying the number of unique browsers ( which may roughly equate to actual unique users ) sending events. This means if the same browser was to send 2 video_view events, they would be recorded as a single unique entity.

The users can be tracked by two means:

  1. A Server side browser cookie
  2. Client side unique identifier


Opt-in

User tracking is enabled only when the cookies_enabled parameter is set to any value other than false. This includes the case where no value is set, i.e., if the player sends an event with cookies_enabled=, we would still not track the user. Also, if the user parameter is specified, we would continue to track the user.


Server side identifier cookie

This is the most common and simplest use case. If the player sends a request to the analytics collector with the track flag set. The server will generate a cookie with name bcau and value set to a UUID. This cookie would be set with a max-age of 365 days after which a new cookie would be set.

  1. The max age of the cookie is reset on every visit to 365 days.
  2. If the cookie tracking is disabled (by sending cookies_enabled=false) the collector would simply not continue to set/reset cookies and will no longer record user tracking information. This implies that the cookies that are already set would continue to exist on the viewer's browser until they are cleared but would not be used by the collection system.


Client side identifier

In certain cases, such as native mobile apps or certain browsers (cookies disabled), it may be difficult or impossible to accept/set/store cookies. In this scenario, the user/client application may create a unique identifier (such as a device UUID) and send it as a the user query parameter.
