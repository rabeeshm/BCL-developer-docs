OAuth2 flows

Audience: OAuth2 client users (callers of Brightcove APIs)

Use this page to determine what flow a client should use to interact with Brightcove APIs.

Client credentials flow

  • I am a script or service that doesn't have a logged-in user (e.g., I run automatically without user intervention, like a cron job). I want to get access to a specific set of resources, without necessarily making the owner of those resources log in every time. ("I am a service developer testing my API" is usually a special case of this.) I am running in a context that is secure enough to hold a long-lived persistent access object for those resources, and I am being run by a person or entity authorized to hold such an object.
    • You want the client credentials flow. A client ID and secret for client credentials flow is as dangerous as a Media API token: it is a complete credential for the resources it grants access to, without any other authenticator required. For many use cases, this is exactly what you need.

Authorization code flow

  • I am a server based application that has a logged-in user (e.g., I am the server side of a webapp and I need to make some calls to another API). I want to do things invoking Brightcove APIs on the user's behalf, but I am not necessarily allowed to handle their actual authenticators (like BC_TOKEN). However, I am not a browser or phone app; I can keep secrets.
    • You want the authorization code flow. (This is often the most appropriate flow for services that want to invoke other services as clients.)
