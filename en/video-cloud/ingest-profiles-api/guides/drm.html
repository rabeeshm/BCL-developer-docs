Content Security (DRM and HLSe)

Video Cloud offers several forms of content security, some of which can be used together. This topic deals with security that can be configured in your ingest profiles:

  • DRM
  • HLS encryption (HLSe)

For additional content security mechanisms that can be set in either the player configuration or video properties, see:

  • Configuring Player Availability
  • Configuring Video Availability

Note: DRM is currently not support on any mobile OS browsers. Widevine is supported for iOS and Android apps.

Note: A DRM player should not be used for iDevices since it would prompt to install SilverLight. Workaround: Two players can be created, one with DRM support and one without, and switch between them based on the platform/video.

DRM

DRM (Digital Rights Management) protects your video content by encrypting the video data and unlocking it based on license policies. When the video content is loaded into a Brightcove player, the player calls back to a licensing server and obtains permission to play the video. This transaction is transparent to viewers.

Note: to use DRM or HLS encryption, your account must be enabled for these features. Contact your account manager for more information.

Video Cloud currently supports the several DRM types for different video formats:

Video Format
MPEG-DASH Smooth HLS Other
DRM type widevine (Modular)
fairplay
marlin
playready
hds-faxes (Adobe Primetime/Access) HDS, RTMP
widevine (Classic) [1-1] WVC

Notes

  • [1-1] Widevine Classic is supported only for Android Apps and is deprecated - you should use Widevine Modular instead.

Reference IDs

Renditions are assigned to DRM packages by reference_id, so you need to be sure that you assign a reference_id to each rendition defined in profiles that will have DRM packages.

Example

..."renditions": [
    {
        "media_type": "video",
        "reference_id": "av0",
        "format": "mp4",
        "audio_codec": "aac",
        "width": 360,
        "height": 270,
        "h264_profile": "baseline"
    }...
]

Note: every reference_id must be unique across all profiles defined for your account.

Defining packages

For each DRM package you want to define, add an object to packages array specifying the package type and an array of rendition reference_ids to include in the package:

"packages" : [
    {
        "package_type" : "widevine",
        "renditions" : [ "wv0", "wv1","wv2", "wv3","wv4", "wv5","wv6", "wv7" ]
    },
    {
        "package_type" : "hds-faxs",
        "renditions" : [ "hds0" ]
    },
    {
        "package_type" : "hds-faxs",
        "renditions" : [ "hds1" ]
    },
    {
        "package_type" : "hds-faxs",
        "renditions" : [ "hds2" ]
    }
]

For MPEG-DASH renditions, the syntax is slightly different, including an array of DRM types and dash as the package type:

"packages" : [
    {
        "drm" : ["widevine", "playready", "marlin"],
        "package_type" : "dash",
        "renditions" : "av01"
    }
]

Note: in the JSON for the profile, the renditions array must be defined before the packages array, so that the reference_ids referred to in the packages exist.

Widevine

Google's Widevine DRM packaging comes in two flavors: Modular, for MPEG-DASH video, and Classic, for WVC video. Modular is the newer type and recommended.

Widevine Modular

Widevine Modular is for MPEG-DASH video.

Notes

  • For MPEG-DASH renditions using Widevine Modular DRM, each rendition must include the property "streaming_delivery_profile": "live". Otherwise, Widevine protection will not be applied to the rendition.

HLS encryption (HLSe)

HLSe is not a DRM packaging mechanism, but encrypts the HLS segments to protect your content. HTTP Live Streaming (HLS) Encryption lets you send encrypted video over HTTP for playback on desktop and mobile devices. It utilizes the Advanced Encryption Standard (AES) as well as other technologies to seamlessly protect content from non-authorized streaming, piracy, and redistribution by others, with no detectable difference to video playback. For more information on HLSe, see Protecting Videos with HLS Encryption.

To add encryption for your HLS renditions, add the encryption_method and encryption_key_rotation_period fields to each of the HLS renditions defined in your ingestion profile: 

{
  "media_type": "video",
  "reference_id": "ts0",
  "format": "ts",
  "type": "segmented",
  "audio_codec": "aac",
  "audio_bitrate": 64,
  "video_codec": "h264",
  "encryption_method": "aes-128",
  "encryption_key_rotation_period": 10,
  "video_bitrate": 450,
  "decoder_bitrate_cap": 771,
  "decoder_buffer_size": 1028,
  "keyframe_rate": 0.5,
  "max_frame_rate": 30,
  "width": 480,
