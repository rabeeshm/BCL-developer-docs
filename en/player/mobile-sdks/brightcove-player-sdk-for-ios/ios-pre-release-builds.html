Change log

Changes in BCOVPlayerSDK v4.1.3 (2014-02-07)

  • Introduced -resumeAd and -pauseAd to BCOVPlaybackController. If an ad plugin is present, these will resume or pause the ads
  • Added BCOVDelegatingSessionConsumer.h to the global BCOVPlayerSDK.h header file.
  • Fixed an issue with Airplay that could cause a video in a playlist to be replaced with the next video in the playlist prematurely.
  • Fixed a crash that could occur after calling -[BCOVPlaybackController setVideos:] a second time.
  • Deprecated -[BCOVPlaybackController seekToTime:]. When seeking, call seek directly on the AVPlayer in the current session.

Changes in BCOVIMA v1.0.3 (2014-02-07)

  • Added the ability to supply companion ad slots using a custom BCOVIMAAdsRequestPolicy by using the newly added +[BCOVIMAAdsRequestPolicy videoPropertiesVMAPAdTagUrlAdsRequestPolicyWithCompanionAdSlots:] and +[BCOVIMAAdsRequestPolicy adsRequestPolicyWithVMAPAdTagUrl:companionAdSlots:]. Simply set the the adsRequestPolicy property on a BCOVIMASessionProviderOptions when creating your IMA session provider.
  • Added a public constant for kBCOVIMAAdTag to BCOVIMAComponent.h.
  • Fixed a crash that could occur when transitioning between videos.
  • Fixed an issue where ads would not be cleaned up when calling -[BCOVPlaybackController setVideos:] a second time.
  • Removed the ability to specify a BCOVIMASessionDeliveryPolicy.

Changes in BCOVWidevine v1.0.3 (2014-02-07)

  • Fixed an issue where Widevine would suffer state corruption when calling -[BCOVPlaybackController setVideos:] a second time.

4.1.2 beta (2014-01-28)

  • Introduce BCOVDelegatingSessionConsumerDelegate helper class. It implements the BCOVPlaybackSessionConsumer protocol, and accepts an Objective-C delegate object, which will be sent messages according to events that transpire on the current playback session. This provides a more traditional alternative to RACSignals on playback session objects for event handling. This new class is intended primarily for more advanced cases, most clients can and should continue to conform to BCOVPlaybackControllerDelegate for this functionality.
  • Change the type of the second parameter to BCOVPlaybackControllerViewStrategy blocks. Previously, the type was id<BCOVPlaybackSessionConsumerContainer>, a protocol which existed only for use in view strategies. This type has been removed entirely and the type is now id<BCOVPlaybackController>
  • Corrected the documented semantics of the -[BCOVPlaybackSession lifecycle] signal. To be clear: this signal only sends values for events that occur after it is subscribed to, though in previous releases the documentation stated that the signal always sends the entire history of events for each subscription.
  • Significant performance improvements to code that detects whether cue points are hit.
  • Fixed a bug that could cause the application to freeze or crash when reaching the end of a playlist.
    • Known issues
    BCOVPlayerSDK v4.1.2
    • BCOVDelegatingSessionConsumerDelegate.h is missing from the BCOVPlayerSDK.h header file. This file will need to be added manually.
    BCOVIMA v1.0.2
    • Using [BCOVIMASessionDeliveryPolicy waitForVMAPResponseDeliveryPolicy] may cause instability when switching between videos with ads. This does not surface when using the default BCOVIMASessionDeliveryPolicy.

4.1.1 beta (2014-01-18)

  • BCOVPlayerSDKManager's API for creating playback controllers has been streamlined, to provide a more straightforward path to create a playback controller.
  • BCOVIMA and BCOVWidevine components are no longer instantiated as objects and registered directly with the BCOVPlayerSDKManager; instead, this happens implicitly when any of the component-specific manager methods are called.
  • The methods added to BCOVPlayerSDKManager in each of the components were streamlined and made consistent with each other. There is now one method in each component to create a component-specific session provider, and any optional configuration of session providers is provided via an "options object". There is also one convenience method in each component for creating a playback controller that is pre-configured for that specific component. (If you want to use both components, you no longer have a convenience method; you need to create separate session providers and then create a playback controller from them. See the next item.)
  • Removed the BCOVPlayerSDKManager convenience method for creating a Widevine + IMA -capable playback controller from the BCOVWidevine component. That method was provided as a temporary shim in the first beta release because we didn't have a solid API for composing multiple session providers together. Now that we have APIs for doing so, it is no longer needed.
